The counter lives inside the compiled module, not in JavaScript.

Zig source

var counter: i32 = 0;

pub export fn add(a: i32, b: i32) i32 {
    return a + b;
}

pub export fn getCounter() i32 {
    return counter;
}

pub export fn incrementCounter() i32 {
    counter += 1;
    return counter;
}

pub export fn resetCounter() void {
    counter = 0;
}

export tells Zig to emit the function as a symbol in the final wasm module so JavaScript can access it through instance.exports. pub is separate: it makes the declaration public to other Zig code.

Build and publish flow

This page is documented around one path only: build a static wasm file for try/, publish that folder, and load ./demo.wasm from the browser. 

const target = b.resolveTargetQuery(.{
    .cpu_arch = .wasm32,
    .os_tag = .freestanding,
});

const root_module = b.createModule(.{
    .root_source_file = b.path("src/main.zig"),
    .target = target,
    .optimize = optimize,
});

const wasm = b.addExecutable(.{
    .name = "demo",
    .root_module = root_module,
});

wasm.entry = .disabled;
wasm.rdynamic = true;

const install_try = b.addInstallArtifact(wasm, .{
    .dest_dir = .{ .override = .{ .custom = "try" } },
});

const wasm_try_step = b.step(
    "wasm-try",
    "Build WebAssembly artifact into zig-out/try for static publishing",
);
wasm_try_step.dependOn(&install_try.step);
  1. Write Zig functions with export fn so they become visible in the final wasm module.
  2. Target wasm32-freestanding to build a small module with no host OS assumptions.
  3. Point root_source_file at your Zig entry source, here src/main.zig.
  4. Disable the normal executable entry point with wasm.entry = .disabled.
  5. Keep exported symbols visible with wasm.rdynamic = true.
  6. Install the artifact into zig-out/try and expose it as zig build wasm-try.

That is the reproducible core. If you copy those pieces into another project, you can produce a browser-loadable wasm file without additional Zig-specific tooling.

  1. Create src/main.zig with exported functions that use simple ABI-friendly types like integers.
  2. Add the wasm target and build step in build.zig.
  3. Run zig build wasm-try to produce zig-out/try/demo.wasm.
  4. Copy that file into your static folder as try/demo.wasm.
  5. Load it from JavaScript with a relative URL based on import.meta.url.
  6. Read the exported functions from instance.exports and wire them into the page UI.

In this demo, bun run build:try runs zig build wasm-try and then copies zig-out/try/demo.wasm into try/demo.wasm so GitHub Pages can publish a fully static folder.

How the page includes wasm

The browser code keeps the loading path explicit and small:

const wasmUrl = new URL("./demo.wasm", import.meta.url);
const response = await fetch(wasmUrl);
const { instance } = await WebAssembly.instantiateStreaming(response, {});
const { add, getCounter, incrementCounter, resetCounter } = instance.exports;

If the server does not send application/wasm, the loader falls back to arrayBuffer() plus WebAssembly.instantiate().
